## 五、不建議當成案件內容移出的項目

下列屬程式運作或固定介面規則，移到 Markdown 反而容易讓非工程人員誤改造成故障：

- 路由：`/`、`/issues`、`/map`、`/faq`、`/plants`。
- Google 地圖座標合法範圍：經度 -180～180、緯度 -90～90。
- 搜尋、篩選、展開 FAQ、畫面動畫及響應式版面行為。
- CSS 顏色、間距、字級、陰影、圖示及等高線繪圖參數。
- localStorage key、自訂瀏覽器事件名稱、React 元件名稱及資料欄位型別。
- 開發環境變數 `GEMINI_API_KEY`、`APP_URL`：這類金鑰或部署網址不應放入可公開提交的 content；而且目前程式是否實際使用仍需確認。

## 六、建議的下一階段最小修改範圍（覆核後才執行）

建議分批進行，避免一次更動大量檔案：

1. **第一批：地圖設定**
   - 新增 `src/content/map.md`。
   - 移出兩張地圖網址、預設座標、縮放層級、標題、版本及更新日期。
   - 只修改地圖資料讀取，不改路由、版面或功能。
2. **第二批：平台基本資料**
   - 新增 `src/content/site.md`。
   - 移出平台名稱、頁尾、AI 網址、首頁卡片文字及背景圖片等經覆核項目。
   - 程式路徑、圖示與樣式仍保留在程式內。
3. **第三批：改善既有案件內容格式**
   - 依覆核決定是否把 `issues.json` 改成易讀 Markdown。
   - 先確認重複設施資料的正式數值，再設計避免重複填寫的格式。
4. 每一批皆執行 TypeScript 檢查、production build 及相關頁面人工檢查後，再分別提交。

### 預期風險

- Markdown 欄位拼錯或漏填，可能造成頁面空白或顯示預設值。
- 地圖完整網址格式若不正確，嵌入地圖可能無法載入。
- 將 JSON 改為 Markdown 時，陣列與多層議題資料需要明確格式，否則容易解析錯誤。
- 平台文字集中後，若不同頁面原本使用不同說法，統一時可能改變目前顯示內容。

### 回滾方式

每一批獨立建立 Git commit；若上線後有問題，使用 `git revert <該批次的 commit hash>` 回復。不要直接刪除整個 `content` 資料夾。
